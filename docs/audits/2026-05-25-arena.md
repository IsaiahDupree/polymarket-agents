## Methodology

Categories scanned in parallel via `Grep`:
1. **Error handling** — swallowed catches, missing try/wrap, missing timeouts
2. **Security** — hardcoded secrets, SQL injection vectors, unsanitized input on API routes
3. **Race conditions** — shared mutable state, check-then-act without atomicity
4. **Resilience** — retries, circuit breakers, unbounded loops/queues, graceful shutdown

Findings ranked critical → low. Each row names a concrete fix.

## Persistent-worker question (separate concern)

The WS worker is currently **foregrounded in the user's interactive session** (background task ID `be2oy64x8`). When the session ends, the process dies and `realtime_ticks` stops populating. Arena scheduled task keeps running every 5 min (separate Windows task), but it stops receiving WS overrides.

Three options ranked by cost / robustness:

| Option | Effort | Robustness |
|---|---|---|
| A — Leave foregrounded | 0 min | Stops when session ends. Fine for "watch overnight, restart manually." |
| B — `install-realtime-worker.ps1` only (mirrors `install-arena-tasks.ps1`) — Windows scheduled task at logon, `MultipleInstances=IgnoreNew` | ~15 min | Survives reboots. Doesn't survive crashes mid-session. |
| **C — Supervised wrapper** (`run-realtime-supervised.ps1`) — parent script that restarts the worker on exit with capped backoff. Schedule THAT at logon. Pair with finding #6 (stale-heartbeat alert) for visibility. | ~45 min | Survives reboots AND crashes. Poor-man's PM2. |
